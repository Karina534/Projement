# Техническое задание: система управления проектами и задачами

## 1. Общая информация

### 1.1. Назначение системы

Система предназначена для управления проектами, задачами и рабочими процессами команды. Пользователь должен иметь возможность зарегистрироваться, создать проект, создать доску проекта, добавлять задачи, назначать исполнителей, менять статусы задач и оставлять комментарии.

### 1.2. Основные сущности системы

В системе используются следующие основные сущности:

- пользователь;
- проект;
- доска проекта;
- колонка доски;
- задача;
- комментарий к задаче.

### 1.3. Основной workflow задач

Для работы с задачами используется канбан-доска со следующими колонками:

- Backlog;
- To Do;
- In Progress;
- Review;
- Done.

---

## 2. Функциональные требования

## 2.1. Регистрация и авторизация пользователей

### Требования

Пользователь должен иметь возможность создать учетную запись и войти в систему по email и паролю.

Email пользователя должен быть уникальным. Пароль должен храниться в защищенном виде, например в виде хэша, а не в открытом тексте.

### Пример сценария

Пользователь открывает страницу регистрации, вводит имя, email и пароль. После успешной регистрации пользователь получает доступ к системе.

### Критерии приемки

- пользователь может зарегистрироваться в системе;
- пользователь может войти в систему по email и паролю;
- при вводе неверных данных система выводит ошибку;
- неавторизованный пользователь не имеет доступа к проектам;
- email пользователя должен быть уникальным;
- пароль не должен храниться в открытом виде.

---

## 2.2. Создание проекта

### Требования

Пользователь должен иметь возможность создать проект, указав название и описание.

Пользователь, создавший проект, автоматически получает права владельца проекта.

### Пример сценария

Пользователь создает проект «Разработка таск-трекера». После создания проект появляется в списке доступных проектов.

### Критерии приемки

- проект создается и сохраняется в системе;
- при пустом названии проекта выводится ошибка;
- проект отображается в списке доступных проектов;
- создатель проекта получает права владельца.

---

## 2.3. Создание доски проекта

### Требования

Внутри проекта должна создаваться доска для работы с задачами.

Доска должна содержать основные колонки workflow:

- Backlog;
- To Do;
- In Progress;
- Review;
- Done.

### Пример сценария

Для проекта создается доска с колонками «Backlog», «To Do», «In Progress», «Review», «Done».

### Критерии приемки

- доска создается внутри проекта;
- колонки доски отображаются пользователю;
- задачи могут быть связаны с конкретной колонкой;
- каждая задача должна относиться к одной из колонок доски.

---

## 2.4. Создание задачи

### Требования

Пользователь должен иметь возможность создать задачу внутри проекта.

У задачи должны быть следующие поля:

- название;
- описание;
- статус;
- исполнитель;
- дедлайн;
- приоритет.

### Пример сценария

Пользователь создает задачу «Реализовать авторизацию», указывает описание, высокий приоритет и срок выполнения.

### Критерии приемки

- задача создается и сохраняется в системе;
- задача отображается на доске проекта;
- при отсутствии названия задачи выводится ошибка;
- задача связана с конкретным проектом;
- задача связана с конкретной колонкой доски.

---

## 2.5. Назначение исполнителя, дедлайна и приоритета

### Требования

Пользователь должен иметь возможность назначить исполнителя задачи, указать срок выполнения и приоритет.

Исполнителем может быть только пользователь, который входит в проект.

### Пример сценария

Менеджер назначает задачу backend-разработчику, устанавливает дедлайн и высокий приоритет.

### Критерии приемки

- исполнитель задачи сохраняется в системе;
- дедлайн задачи сохраняется в системе;
- приоритет задачи сохраняется в системе;
- исполнитель, дедлайн и приоритет отображаются в карточке задачи;
- нельзя назначить исполнителем пользователя, который не входит в проект.

---

## 2.6. Изменение статуса задачи

### Требования

Пользователь должен иметь возможность менять статус задачи в соответствии с этапами workflow.

Статус задачи должен соответствовать одной из колонок доски.

### Пример сценария

Пользователь переводит задачу из статуса «To Do» в статус «In Progress».

### Критерии приемки

- статус задачи изменяется и сохраняется;
- после обновления страницы задача остается в новом статусе;
- изменение статуса отображается на доске;
- при смене статуса задача перемещается в соответствующую колонку.

---

## 2.7. Отображение канбан-доски

### Требования

Задачи проекта должны отображаться на канбан-доске по колонкам workflow.

Внутри каждой колонки должны отображаться карточки задач, относящиеся к соответствующему статусу.

### Пример сценария

На странице доски отображаются колонки «Backlog», «To Do», «In Progress», «Review», «Done». Внутри колонок находятся карточки задач.

### Критерии приемки

- задачи отображаются в соответствующих колонках;
- пользователь видит название задачи;
- пользователь видит исполнителя задачи;
- пользователь видит приоритет задачи;
- пользователь видит дедлайн задачи;
- при изменении статуса задача переходит в другую колонку.

---

## 2.8. Добавление комментариев к задаче

### Требования

Пользователь должен иметь возможность оставить комментарий в карточке задачи.

Комментарий должен быть связан с конкретной задачей и конкретным автором.

### Пример сценария

Исполнитель пишет комментарий: «Начал выполнение задачи, требуется уточнить требования».

### Критерии приемки

- комментарий сохраняется в системе;
- комментарий отображается в карточке задачи;
- у комментария отображается автор;
- у комментария отображается дата создания;
- пустой комментарий нельзя отправить.

---

## 3. Роли пользователей

В системе предусмотрены следующие роли пользователей:

### 3.1. Неавторизованный пользователь

Неавторизованный пользователь может только зарегистрироваться или войти в систему. Он не имеет доступа к проектам, доскам, задачам и комментариям.

### 3.2. Участник проекта

Участник проекта может просматривать проект, доску и задачи, к которым у него есть доступ. Также он может оставлять комментарии к задачам и изменять данные задач в рамках выданных прав.

### 3.3. Исполнитель задачи

Исполнитель задачи — это участник проекта, назначенный ответственным за выполнение конкретной задачи. Он отображается в карточке задачи.

### 3.4. Владелец проекта

Владелец проекта — это пользователь, создавший проект. Он имеет права на управление проектом, доской, задачами и участниками проекта.

## 4. Требования к валидации данных

Система должна проверять корректность пользовательского ввода.

Обязательные проверки:

- email пользователя должен быть уникальным;
- пароль не должен быть пустым;
- название проекта не должно быть пустым;
- название задачи не должно быть пустым;
- комментарий не должен быть пустым;
- исполнитель задачи должен входить в проект;
- статус задачи должен соответствовать одной из колонок доски.

---

## 5. Нефункциональные требования

### 5.1. Безопасность

- пароли должны храниться в защищенном виде;
- неавторизованные пользователи не должны иметь доступ к закрытым данным;
- пользователь не должен иметь доступ к проектам, в которых он не участвует.

### 5.2. Надежность хранения данных

- созданные проекты должны сохраняться в системе;
- созданные задачи должны сохраняться в системе;
- изменения статусов задач должны сохраняться;
- комментарии должны сохраняться;
- после обновления страницы данные должны оставаться актуальными.

### 5.3. Удобство использования

- пользователь должен видеть список доступных проектов;
- пользователь должен видеть задачи на канбан-доске;
- пользователь должен понимать текущий статус задачи;
- ошибки валидации должны отображаться понятным образом.

---

## 6. Итоговые критерии готовности

Функциональность считается реализованной, если:

- пользователь может зарегистрироваться и войти в систему;
- пользователь может создать проект;
- внутри проекта создается доска с колонками workflow;
- пользователь может создавать задачи;
- пользователь может назначать исполнителя, дедлайн и приоритет;
- пользователь может менять статус задачи;
- задачи отображаются на канбан-доске;
- пользователь может добавлять комментарии к задачам;
- система корректно обрабатывает ошибки валидации;
- неавторизованные пользователи не имеют доступа к проектам.
